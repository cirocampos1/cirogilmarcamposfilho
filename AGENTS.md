## knowledge architecture

Before any non-trivial task, consult `ORQUESTRA.md` and follow its governance, Spec-Driven Development, WOW/design, validation, branch, context, commit, and deployment rules. Do not wait for the user to mention the guide explicitly.

This project uses Graphify as its core knowledge graph and context intelligence engine.

**Token Optimization (RTK):**
O ambiente possui o RTK (Rust Token Killer) instalado e configurado globalmente. Comandos de terminal como `git`, `ls`, `grep` e test runners são interceptados e comprimidos automaticamente, garantindo extrema economia de tokens (60-90%). Os agentes podem e devem executar comandos shell rotineiros com a tranquilidade de que o output será devolvido já formatado e enxuto.

Rules:
- Before any architectural analysis or multi-file change, you MUST run `graphify query` or `graphify explain` to assess risk and understand the context.
- Use `graphify query` for semantic search across the codebase instead of multiple grep calls.
- After significant code changes, run `graphify update .` in the terminal to keep the knowledge graph synchronized.
- In this Windows/PowerShell workspace, use the CLI form `graphify ...`; `/graphify ...` is slash-command documentation and is not a valid terminal command here.

## frontend visual validation

When changing or fixing visual UI code (templates, CSS, browser JavaScript, page layout, modals, dashboards, Kanban/wallboard screens):
- Use Playwright with the local config file `.env.playwright`; it contains `_PLAYWRIGHT_BASE_URL` and the optional login settings
- Capture/check desktop and mobile screenshots for the affected page when feasible
- Prefer focused routes via `_PLAYWRIGHT_ROUTES` instead of running unrelated screens
- Use `_PLAYWRIGHT_USERNAME` and `_PLAYWRIGHT_PASSWORD` for authenticated screens when credentials are available in the environment or provided by the user
- Review screenshots for obvious regressions: overflow, broken spacing, invisible text, contrast issues, clipped modals, sidebar/header overlap, mobile breakage, and loading/error states
- If Playwright cannot run, record the blocker and use the best available fallback validation
- Do not commit generated Playwright artifacts; they belong under ignored `artifacts/playwright/`

## graphify

This project has a knowledge graph at graphify-out/ with god nodes, community structure, and cross-file relationships.

When the user types `/graphify`, invoke the `skill` tool with `skill: "graphify"` before doing anything else.

Rules:
- For codebase questions, first run `graphify query "<question>"` when graphify-out/graph.json exists. Use `graphify path "<A>" "<B>"` for relationships and `graphify explain "<concept>"` for focused concepts. These return a scoped subgraph, usually much smaller than GRAPH_REPORT.md or raw grep output.
- Dirty graphify-out/ files are expected after hooks or incremental updates; dirty graph files are not a reason to skip graphify. Only skip graphify if the task is about stale or incorrect graph output, or the user explicitly says not to use it.
- If graphify-out/wiki/index.md exists, use it for broad navigation instead of raw source browsing.
- Read graphify-out/GRAPH_REPORT.md only for broad architecture review or when query/path/explain do not surface enough context.
- After modifying code, run `graphify update .` to keep the graph current (AST-only, no API cost).
